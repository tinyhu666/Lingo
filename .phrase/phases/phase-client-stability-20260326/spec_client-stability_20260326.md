# Client Stability Spec

## Summary

本阶段聚焦桌面客户端稳定性与体感优化，优先解决窗口四角圆角重叠、翻译触发链路冗余与设置即时生效问题，同时修复当前主界面中能直接影响使用的 UI/功能缺陷。

## Goals

- 桌面端窗口四角仅保留单一、干净的圆角外观。
- 翻译快捷键触发链路减少不必要的系统操作并提升稳定性。
- 用户在主页/翻译页调整配置后，后续翻译能立即使用最新配置。
- 相同翻译请求在短时间内尽量复用代理热路径，失败时保留足够诊断信息。
- 翻译未完成时重复触发应给出明确反馈，瞬时代理抖动应尽量自动恢复一次。
- 桌面端首次翻译的冷启动成本应尽量在后台预热阶段提前消化。
- 未命中缓存的新文本请求也应尽量通过更短 prompt 和更小生成预算缩短首包时延。
- 满足条件的短文本请求应优先走快模型，且快模型抖动时能自动回退到主模型而不影响后续重试。
- 桌面端开发态调试时也应尽量复用真实翻译代理链路，避免本机验证路径与发布行为脱节。
- 已下线的内部或历史页面不应因为共享导航回归而重新暴露给桌面端用户。
- Windows 桌面端的外层窗口四角不能再叠加前端壳层圆角，必须只保留单一外轮廓。
- 开发者在本机运行桌面端时，应尽量一次命令就拉起完整调试链路，而不是手工拼装多个依赖进程。
- 客户端翻译链路至少需要最小自动化测试覆盖，避免每次都靠人工启动桌面端回归。
- 热键复制、probe 过滤和回填路径中的基础判断也需要最小自动化覆盖，避免系统交互层之下的纯逻辑反复回归。
- 准备正式发版时，前端、Tauri 与 Cargo 的版本元数据需要保持一致，并能产出对应版本号的本地安装包。

## Non-goals

- 不重做整套视觉风格。
- 不引入需要额外后端配置的新能力。

## User Flows

- 用户打开桌面客户端时，窗口四角呈现单层圆角，无重叠边线或背景残影。
- 用户修改翻译语言或风格后，下一次快捷键翻译立即按最新设置执行。
- 用户在游戏中触发翻译快捷键时，客户端尽量减少额外粘贴占位文字等打断动作。
- 用户短时间重复翻译相同内容时，代理优先复用热缓存或共享正在进行的模型请求。
- 用户在上一条翻译尚未结束时再次按下快捷键，会收到“仍在处理中”的提示而不是静默无反应。
- 用户打开客户端或重新启用翻译后，应用会在后台静默预热翻译代理，尽量缩短第一条翻译的等待。
- 用户发送全新文本时，代理会按文本长度和翻译/润色意图自适应控制请求预算，而不是始终走最大生成配置。
- 用户发送满足条件的短文本时，代理会优先尝试快模型；若快模型瞬时失败，本次请求自动回退主模型，但下一次仍可继续尝试快模型。
- 开发者在本机启动桌面客户端且未显式配置后端地址时，客户端会默认尝试本地翻译代理，并在代理未启动时给出明确提示。
- 桌面端用户打开侧边栏时，只会看到当前仍被支持的导航项，不会看到已经下线的 Settings 入口。
- 开发者执行 `tauri dev` 时，Vite 与本地翻译代理会自动一起启动，桌面端预热无需额外手动准备。
- 开发者执行 Rust 测试时，可以直接验证客户端翻译请求能打到本地假代理，并验证本地代理不可达时的错误提示。
- 开发者执行 Rust 测试时，也能验证 `daily_mode` 与普通模式的复制/粘贴快捷键路径，以及 probe 过滤逻辑是否仍然正确。
- Windows 桌面端用户打开客户端时，原生窗口外轮廓是唯一的四角圆角来源，前端壳层不再额外收圆。
- 发布者同步到 `0.4.0` 后执行完整验证与本地打包时，可以直接得到对应版本号的 macOS 安装包和 updater 签名文件。

## Edge Cases

- Tauri 预览环境和浏览器预览环境都应保持可用。
- 剪贴板在复制后未立即更新时，客户端应尽量等待正确内容而非直接读取旧值。
- 代理出现一次性 5xx/超时时，客户端应优先做窄范围自恢复而不是立刻把抖动暴露给用户。
- 设置加载失败时不能阻塞 UI 基本渲染。
- 代理热缓存不能跨模型、温度或翻译方向错误复用。
- 预热请求不能造成用户可见报错，也不能在短时间内重复打满后台。
- 自适应预算不能过度压缩，导致正常翻译结果被截断或风格明显劣化。
- 快模型回退不能把主模型结果写入快路由缓存，否则会掩盖后续恢复。
- 开发态默认本地代理不能影响发布构建的后端配置优先级。
- 留在代码库中的历史页面文件不能被误认为“应该重新接回导航”。
- Windows 与 macOS 的外层圆角策略可以不同，但都只能保留一个可见外轮廓。
- 开发态编排脚本不能造成子进程残留，也不能改变既有 `npm run dev` 的单独前端开发入口。
- 测试中的环境变量改写和本地端口占用不能互相污染，否则会让客户端链路测试变得不稳定。
- 即使 OS 级热键和真实输入框回填仍需手工验证，底层的纯逻辑判断也应先被自动化测试拦住。
- 本地打包可以验证版本同步与 macOS 产物链路，但不能替代 tag 推送后由 GitHub Actions 产出的 Windows 安装包与 `latest.json`。

## Acceptance Criteria

- 桌面端圆角不再出现多层重叠视觉。
- `npm run build` 与 `cargo check` 通过。
- `server/translate-proxy` 的 smoke 测试通过。
- 关键设置更新与翻译链路的代码路径可从文档与变更记录追溯。
- 重复触发翻译时客户端会给出忙碌提示，且不会因为读取旧剪贴板造成误翻。
- 启动或重新启用应用后会静默触发一次可控频率的翻译代理预热。
- 代理响应会回传实际生效的 prompt 类型与生成预算，便于定位首发慢请求。
- 代理响应会明确返回 `model_route`，并且在仅配置 fast lane key 的条件下仍可完成符合条件的快路由翻译。
- 开发态同时运行本地代理和 Tauri 客户端时，启动预热日志应显示真实命中本地代理而不是因为缺少后端地址直接失败。
- 桌面端侧边栏不再展示 Settings 入口，且共享页面映射不再包含该项。
- Windows 桌面端外层四角不再出现双层或重叠圆角，macOS 当前正常表现不被回归破坏。
- `npm run tauri dev -- --no-watch` 会自动拉起 Vite 和本地代理，并在启动日志中看到预热成功命中本地代理。
- `cargo test --manifest-path src-tauri/Cargo.toml -- --nocapture` 至少会通过两条客户端翻译链路测试：真实翻译请求 smoke 和本地代理不可达提示。
- `cargo test --manifest-path src-tauri/Cargo.toml -- --nocapture` 还会通过 `shell_helper` 的快捷键路径和剪贴板 probe 过滤测试。
- `package.json`、`package-lock.json`、`src-tauri/Cargo.toml`、`src-tauri/Cargo.lock` 与 `src-tauri/tauri.conf.json` 的版本号需同步到 `0.4.0`，并通过 `npm run build:mac-arm` 产出 `Lingo_0.4.0_aarch64.dmg` 与 `Lingo.app.tar.gz.sig`。
